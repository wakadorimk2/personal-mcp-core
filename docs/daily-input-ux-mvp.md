# daily input UX — MVP 方針定義

> スコープ: daily use UI の主導線 UX を最小仕様として定義する
> 関連 Issue: #176
>
> **この文書は実装仕様ではなく方針の記録である。**
> 後続実装 Issue はこの文書を参照し、整合する範囲で実装すること。

---

## 1. 主導線の配置方針

### 1.1 レイアウト原則

```
[ 草ヒートマップ（28日分）]   ← 上部固定
[ 候補タグ（ワンタップ）     ]   ← ヒートマップ直下
[ 自由記述入力フィールド     ]   ← 候補タグ直下
```

- **草ヒートマップを最上部に固定する。** 過去28日の記録状況を常に視界に入れる。
- **入力導線はヒートマップの直下に配置する。** スクロールや画面遷移なしに入力できることを優先する。
- 候補タグ → 自由記述の順に並べることで「タップで済む→無ければ書く」という自然なフォールバックを実現する。

### 1.2 設計上の制約

- ヒートマップはオレンジ系で表現し、活動の有無を一目で把握できる形式を維持する（詳細な色分けや段階表示は本 Issue のスコープ外）。
- UI polish（アニメーション・レスポンシブ最適化・フォントサイズ等）はスコープ外とする。

## 2. 候補タグと自由記述の併用方針

### 2.1 基本方針

- **ワンタップ候補と自由記述を常に同一画面に並置する。**
- 候補タグをタップすると、そのテキストを入力値として即時確定（または入力欄に流し込む）できる。
- 候補が適切でない場合でも、自由記述フィールドから入力を継続できる（フォールバック）。
- 候補とテキスト入力は排他ではない。候補タップ後に自由記述を追記・編集してもよい。

### 2.2 失敗時のフォールバック

候補取得が失敗した場合（API エラー、データなし、cold start 直後等）:

- **候補タグ欄は非表示にせず、固定候補のみを表示する。**
- 固定候補が空の場合は候補タグ欄ごと非表示とし、自由記述フィールドのみを表示する。
- エラーメッセージや「候補なし」のラベルは表示しない（ユーザーに余計な認知負荷をかけない）。

## 3. 候補生成の最小仕様（temporal locality ルール）

### 3.1 候補ソース（4系統）

| ソース | 定義 | 特性 |
|---|---|---|
| `recent` | 直近のイベント入力から抽出したテキスト（最新N件程度） | 最も時間的に近い |
| `today_frequent` | 当日すでに入力されたテキストの頻出順 | 当日の文脈に強く依存 |
| `7d_frequent` | 過去7日間の入力テキストの頻出順 | 週単位の生活パターンを反映 |
| `fixed` | コード/設定で定義された固定候補 | 常に安定して存在する |

### 3.2 優先順（マージルール）

```
recent > today_frequent > 7d_frequent > fixed
```

- 4ソースからそれぞれ候補を取得し、上記優先順でリストを構築する。
- 同じテキスト（正規化後）が複数ソースに存在する場合、**より優先度の高いソースの1件のみを残す**（重複排除）。
- 最終的な表示件数上限は **8件** とする。8件に満たない場合はそのまま表示する。

### 3.3 正規化の原則

- 重複排除の比較単位は「表示テキストの完全一致」とする（MVP では正規化は最小限）。
- 大文字/小文字の統一・前後空白の除去程度を正規化範囲とする。

### 3.4 ソース別の抽出単位（最小仕様）

| ソース | 抽出対象 | 抽出単位 |
|---|---|---|
| `recent` | 全ドメインのイベント `data.text` | 直近10件のイベントから文ごとに最大2候補 |
| `today_frequent` | 当日のイベント `data.text` | 当日全件で出現頻度を集計し、文ごとに最大2候補 |
| `7d_frequent` | 過去7日のイベント `data.text` | 7日全件で出現頻度を集計し、文ごとに最大2候補 |
| `fixed` | 設定または定数として定義 | 全件を優先順最下位として追加 |

> `data.text` は Event Contract v1 準拠（`docs/event-contract-v1.md`）。
> legacy record の扱いは reader tolerance（正規化読み取り）に従う。

## 4. cold start から生活語彙への移行方針

### 4.1 cold start 定義

- ローカルの event 記録件数が **一定件数未満（目安: 7件）** の状態を cold start とみなす。
- cold start 判定は API 側で行い、UI 側に判定ロジックを持たせない。

### 4.2 cold start 時の挙動

- `recent` / `today_frequent` / `7d_frequent` のソースからは候補を生成しない（または空として扱う）。
- **固定候補（`fixed`）のみを表示する。**
- 固定候補は日常的な状態変化を表す語彙（例: 「作業開始」「休憩」「移動」等）を中心に構成する。

### 4.3 移行フェーズ（生活語彙への寄せ方）

- event 記録が一定件数以上（目安: 7件）になった時点で、`recent` / `today_frequent` / `7d_frequent` を有効化する。
- 有効化後は `recent > today_frequent > 7d_frequent > fixed` の優先順を適用する。
- 移行の閾値（7件）は固定値として実装し、設定可能にしない（MVP 期間中）。
- 生活語彙への「寄せ」は徐々に起こるのではなく、閾値を超えた時点でソース構成が切り替わる形とする（シンプルに保つ）。

### 4.4 固定候補の管理

- 固定候補はコードまたは設定ファイル（JSON/YAML）で管理する（DB は不要）。
- 初期セットは最小限（8件以下）とし、拡張は後続 Issue に委ねる。
- 固定候補は「生活の基本語彙」を前提とし、ドメイン固有語・略語・敬語は含めない。

## 5. 後続実装への受け皿整理

この方針定義を受けて、以下の観点ごとに後続 Issue を起票することを推奨する。

### 5.1 API 観点

- 候補生成エンドポイントの設計（`GET /api/candidates` 相当）
- 4ソースのクエリロジック実装（JSONL reader を通じた集計）
- cold start 判定ロジックの実装
- 重複排除・優先順マージの実装

#### 5.1.1 候補生成 API の最小 contract

MVP の候補生成 API contract はこの文書に吸収して管理する。

- Endpoint: `GET /api/candidates`
- Response: `200` で候補配列を返す。最大 8 件
- Field:
  - `text`: UI に表示する候補テキスト
  - `source`: 候補ソース（`recent` / `today_frequent` / `7d_frequent` / `fixed`）
- cold start 判定は API 側で実施する
  - 対象イベント件数 `< 7`: `fixed` のみ返す
  - 対象イベント件数 `>= 7`: 4 ソースを有効化する
- マージ優先順: `recent > today_frequent > 7d_frequent > fixed`
- 正規化後テキスト重複は 1 件に統合し、より高優先度ソースを採用する
  - 正規化は `strip + lower`（前後空白除去 + 小文字化）
- 各 `data.text` からは 1 文あたり最大 2 候補まで返す
- 候補対象は `domain != summary` かつ `kind != interaction` のイベントに限定する

```json
[
  { "text": "休憩", "source": "recent" },
  { "text": "作業開始", "source": "fixed" }
]
```

### 5.2 UI 観点

- ヒートマップ上部固定レイアウトの実装（既存ヒートマップコンポーネントの配置見直し）
- 候補タグコンポーネントの実装（タップ→即時入力）
- 自由記述フィールドとの連携（候補タップ後の編集フロー）
- 候補取得失敗時の UI 挙動（フォールバック表示）

### 5.3 テスト観点

- 候補マージ・重複排除ロジックのユニットテスト
- cold start 判定のユニットテスト
- 4ソース各抽出ロジックのユニットテスト
- 表示上限（8件）の境界値テスト

## 5.4 吸収した historical notes

この文書には、独立 issue memo だった UI 方針を吸収している。

Issue #187 由来の決定:

- `/` を主導線とし、ヒートマップ + 候補タグ + 入力欄を同一画面に置く
- `/dashboard` は統合画面への alias として扱う
- 候補タップ後は入力欄へ流し込み、追記・編集を許容する

Issue #213 由来の摩擦改善:

- 保存 CTA は現在の入力内容を反映してよい
- composer の操作面はモバイル下端でも押しやすさを落とさない
- キーボード完了フローは短く保つ

これらは独立 record ではなく、daily input UX の一部として読む。

## 6. 非スコープ（この方針定義でやらないこと）

- 高度な LLM 推論・意味ベースのレコメンデーション
- 完璧なレコメンド最適化（CTR 最適化、A/B テスト等）
- 細かな UI polish（アニメーション、レスポンシブ完全対応）
- 情報設計全体の全面改修
- モバイル/デスクトップ同時完全最適化
- 候補テキストの自動要約・変形
- ユーザーへの行動促進（通知・ストリーク・カウントダウン等）

## 7. Assumptions（未確定事項）

| # | 仮定内容 | リスク | 解消方法の候補 |
|---|---|---|---|
| A1 | `data.text` に複数の有用語が含まれる場合の候補上限と順序付けをどう決めるか | 候補数が増えてノイズ化する可能性 | MVP では 1文あたり最大2候補・文中の自然順・同一文内重複排除で扱う |
| A2 | cold start 閾値を7件と想定したが、実運用上の適切値は不明 | 閾値が低すぎると候補品質が低い状態が長引く | 後続 Issue で実測値を元に調整 |
| A3 | 固定候補の初期セットは未定（語彙の選定が必要） | cold start 体験が固定候補の質に依存する | 後続 Issue で固定候補セットを定義する |
| A4 | 候補タップ時に「確定」か「入力欄流し込み」かの動作が未定 | UX に影響 | MVP では「入力欄流し込み（編集可）」を推奨とする |
| A5 | 候補取得 API の実装形式（MCP tool / HTTP / CLI）が未決定 | UI との連携方法に影響 | API 観点の後続 Issue で決定する |
| A6 | ヒートマップは既存実装（Issue #161）を前提とするが、上部固定への配置変更が既存コンポーネントと競合するかは未確認 | 実装コスト増 | UI 観点の後続 Issue で調査・確認する |
