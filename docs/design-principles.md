> このドキュメントは personal-mcp-core の設計思想を「architecture north star」として明文化したものである。
> 将来の実装判断・設計変更の際に立ち返る基準点として機能することを目的とする。
>
> **配置方針**: このファイル（`docs/design-principles.md`）が詳細版の正本。README に要約 + このファイルへのリンクを保持する。

## 設計原則（8原則）

### 原則 1: 人生モデル — 時間は状態変化の連続である

人生を「時間の流れ」として捉えるのではなく、**「状態変化の記録」として捉える**。
特定の瞬間の活動よりも、「何が変わったか」を記録の単位とする。

> **実装への含意**: イベントは時間の刻みではなく、状態変化の境界に対応して記録される。分単位の計測は不要。

---

### 原則 2: システム目的 — Human Observability 基盤

このシステムの目的は、生産性向上・自己最適化ではなく、**「人が自分の状態を観測できる基盤を提供すること」** である。

- 観測は評価でも最適化でもない
- データを持つ人間が洞察を得るための素材を保持する
- AI が代わりに解釈を行うのではなく、人間が自分で意味を見出せる構造を作る

---

### 原則 3: イベント設計 — 観測可能な事実のみ保持する

イベントは **観測可能な事実** のみを含む。解釈・評価・感情付与は含まない。

- `data.text` には「何が起きたか」を記録し、「それが良かったか悪かったか」は含めない
- スコアや評価値はイベント層には存在しない
- 事実を後から意味づけることは Annotation / Interpretation 層の役割である

---

### 原則 4: イベント境界 — 時間ではなく状態変化で区切る

イベントの区切り単位は **時間の経過** ではなく、**状態の変化** である。

- 「30分作業した」ではなく「作業を開始した」「方針が変わった」「完了した」
- 1つのイベントは1つの状態変化に対応する
- `kind` フィールド（`note` / `session` / `milestone` など）は状態変化の性質を表す分類である

---

### 原則 5: 3層構造 — Event / Annotation / Interpretation

システムは論理的に以下の3層で構成される:

| 層 | 役割 | 特性 |
|---|---|---|
| **Event** | 観測された事実の記録 | 不変・append-only・解釈を含まない |
| **Annotation** | イベントへの注釈・補足 | 後付け可能・元イベントを変更しない |
| **Interpretation** | パターン・意味の解釈 | 主観的・変更可能・上位層の責務 |

**責務境界**:

- Event 層はイベントの事実記録のみを行う。イベントを遡及修正しない
- Annotation 層はイベントへの文脈付与を担う。元のイベントを書き換えない
- Interpretation 層は意味の付与・パターン認識を担う。Event 層のフォーマットに依存しない

> **実装注記**: Annotation / Interpretation の実装方式はこのドキュメントのスコープ外である。この原則は方針の明文化であり、実装の確定ではない。

---

### 原則 6: 役割分担 — 外部サービスがログ生成、core は統合観測レイヤ

- **外部サービス**（PoE2 Client.txt、Git、各種ツール）が activity ログの生成源である
- **personal-mcp-core** はそれらを共通イベント形式に統合し、観測可能な状態に保つ統合観測レイヤである
- core 自体はユーザーを監視・分析しない。素材の保管庫として機能する

> PoE2 はその代表ユースケースとして位置づけられ、core の設計原則はドメイン非依存である。

---

### 原則 7: 解釈の扱い — 意味づけは上位層へ分離する

**基盤（Event 層）は観測事実のみを保持し、意味づけは上位層（Annotation / Interpretation）に委ねる**。

- Event 層に「良い」「悪い」「改善が必要」という情報は含まれない
- AI・ユーザー・外部ツールが解釈を行う場合も、元のイベントは変更しない
- 解釈は時とともに変わりうるが、事実記録は変わらない（append-only 原則と対応）

---

### 原則 8: 最終目的 — 観測 → パターン → 理解 → 知恵

このシステムが目指す状態変化のパス:

```
観測（Observation）  <- システムが提供する基盤
    | 事実が積み重なる
パターン（Pattern）
    | 傾向が見えてくる
理解（Understanding）
    | 自分の状態を把握できる
知恵（Wisdom）       <- 人間が辿り着く
```

このパスは **人間が主体** で辿るものである。システムは「観測」段階の基盤を提供し、パターン以降は人間の判断に委ねる。

---

## 分離方針の明文化

> **「基盤は観測事実、意味づけは上位層」**

この原則はシステム全体を貫く最重要方針である。

| 層 | 含んでよいもの | 含んではいけないもの |
|---|---|---|
| Event | 事実・タイムスタンプ・ドメイン・種別・テキスト | 評価・スコア・感情・解釈 |
| Annotation | 文脈・補足・タグ・関連付け | 元イベントの書き換え |
| Interpretation | 意味・パターン・推論 | Event 層フォーマットへの介入 |
