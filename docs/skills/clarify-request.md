# clarify-request

**種類**: clarify
**正本**: このファイル（`docs/skills/clarify-request.md`）
**Claude用アダプタ**: `.claude/skills/clarify-request/SKILL.md`
**Codex用アダプタ**: `.codex/skills/clarify-request/SKILL.md`

---

## Mission

Issue 作成前に人間への質問のみを行い、目的・制約・優先順位を確定する。
推測で空欄を埋めず、回答不足のまま先に進まない。

---

## Rules（絶対）

- **質問のみ行う** - 回答・提案・実装案・Issue 本文の草案を出さない
- **推測で埋めない** - 情報が不足している項目は TBD と明記し、再質問する
- **未確定項目は再質問する** - 1回の回答で全項目が埋まらない場合は次ラウンドの質問を出す
- **結論を先出ししない** - 「〜すべき」「〜が正解」等の断定表現を使わない
- **実装案を先出ししない** - 技術選択・設計案は clarify フェーズでは提示しない
- **Issue 本文を断定生成しない** - 構造化メモは draft としてラベルし、人間の確認前に確定扱いしない

---

## Inputs

- 要求の断片（一言・口頭メモ・Slack スレッド等）
- 任意: 既存 Issue / PR / ドキュメントへの参照

---

## Procedure（作業手順）

1. **受け取った要求を 1〜2 行で要約する** - ただし断定しない（「〜と理解しました、合っていますか？」）
2. **Clarify Questions テンプレートを出力する** - 未記入の項目はすべて空欄のまま質問として出す
3. **回答を受け取り、埋まった項目を記録する** - 未回答・曖昧な項目は再質問リストに移す
4. **全項目が確定または明示的に TBD と合意されたら** - 構造化メモ（draft）を出力する
5. **構造化メモを人間が確認したら** - draft フェーズ（Issue 草稿作成）へ渡す

---

## Output A: Clarify Questions テンプレート（コピペ用）

> このテンプレートをそのままチャットに貼り付けて回答を埋める。
> 回答できない項目は「TBD」または「不要（理由: ）」と書く。

```text
## Clarify Questions

**Goal**
- このタスクで解決したい問題・達成したいことは何ですか？
- 解決後の「完了」はどんな状態ですか？

**Scope - in**
- 今回の対象に含めるものは何ですか？（機能・ファイル・ドメイン等）

**Scope - out**
- 今回の対象に含めないものは何ですか？（明示しておきたい除外）

**Acceptance Criteria**
- 完了の判定条件を箇条書きで教えてください（テスト・確認手順・成果物など）

**Constraints**
- 技術・運用・コスト・互換性などの制約はありますか？

**Priority / Deadline**
- 優先度（高/中/低）と期日または目安はありますか？

**Dependencies / Blockers**
- 先行して完了が必要な Issue・PR・外部要因はありますか？
- 現時点でブロックされている理由はありますか？
```

---

## Output B: 構造化メモ（draft フェーズへ渡す形式）

> 全項目が確定 or TBD 合意後に出力する。このメモは draft であり、確定 Issue 本文ではない。

```text
## Clarify Result [DRAFT - 人間確認前に確定扱いしないこと]

**Goal**: <確定した目的>

**Scope**
- in: <対象>
- out: <除外>

**Acceptance Criteria**:
- <条件1>
- <条件2>

**Constraints**: <制約（なければ「なし」）>

**Priority**: <高/中/低>
**Deadline**: <日付または「TBD」>

**Dependencies**: <Issue番号・PR・外部要因（なければ「なし」）>
**Blockers**: <現在のブロック要因（なければ「なし」）>

**TBD 項目** (再確認が必要):
- <未確定項目A>: <なぜ未確定か>
- <未確定項目B>: <なぜ未確定か>
```

---

## 回答不足時の扱い

| 状況 | 対応 |
|---|---|
| 項目が空欄のまま | TBD として記録し、次ラウンドで再質問する |
| 回答が曖昧・矛盾 | 曖昧な点を引用して具体化を問い返す |
| 人間が「とりあえず進めて」と言った場合 | TBD 項目を明記した上で draft メモを出力し、確認を促す |
| 全項目 TBD のまま先に進むよう強く求められた場合 | TBD 数を明示して「以下が未確定のまま進めると Issue 品質が下がる」と警告し、人間判断を仰ぐ |

---

## 禁止

- 質問せずに推測で項目を埋める
- 結論・推奨・技術選択を clarify フェーズで先出しする
- 実装案・設計案をこのフェーズで提示する
- 構造化メモを確定 Issue 本文として断定的に出力する
- 人間の回答なしに次フェーズへ自動移行する
