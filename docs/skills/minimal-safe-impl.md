# minimal-safe-impl

**種類**: impl

**Claude用アダプタ**: `.claude/skills/minimal-safe-impl/SKILL.md`

---

## Mission

Issue または実装依頼を、repo の MVP 互換性ポリシーに従って最小差分で実装する。
既存の構造・CLI パターンに合わせ、推測ベースの整理や拡張を入れない。

---

## Rules（絶対）

- 変更は Goal / Scope / 完了条件を満たす最小限に留める
- 既存のディレクトリ構造・責務分割・CLI の命名/引数パターンを踏襲する
- Scope 外のリファクタ・命名整理・抽象化・共通化をしない
- 「あとで必要かもしれない」互換レイヤ・設定・拡張ポイントを追加しない
- 互換性は repo の MVP 方針に従う
  - 保証対象として暗黙に守るのは JSONL イベント形式のみ
  - CLI / 内部モジュール / MCP IF / 設定形式の互換維持を前提にしない
  - データ形式を壊す変更が必要なら `schema_version` とワンタイム移行スクリプトを同伴し、恒久互換レイヤは作らない
- 破壊的変更やデータ変更を伴う場合は、その影響を明記する
- 情報不足では止まらず、妥当な仮定を短く明記して進める
  - ただし破壊的変更になり得る場合は中断して理由を述べる

参照:
- `README.md` の「互換性ポリシー（MVP期間中）」
- `AI_GUIDE.md` の「互換性に関するガードレール」

## Inputs

- Issue番号 / URL / 実装依頼文
- Goal / Scope / 完了条件
- 影響しうる CLI / データ形式 / モジュール

---

## Procedure（作業手順）

1. **要求を要約する** — Goal / Scope / 完了条件 / 仮定を 3〜6 行で整理する
2. **既存パターンを確認する** — 近い実装を読み、構造と CLI パターンを合わせる
3. **最小変更で実装する** — 必要なファイルだけを編集し、挙動を満たす
4. **互換性影響を確認する** — JSONL 形式変更の有無、移行要否、不要な互換レイヤ混入の有無を確認する
5. **最小テスト/検証を実行する** — 変更箇所に最も近いコマンドやテストだけを回す
6. **結果を整える** — 下記 Output テンプレに従って報告する

## 禁止

- Scope 外ファイルへの変更
- 無関係な rename / formatting-only 修正 / import 整理
- 互換レイヤの常設
- 将来用途のための抽象化・設定追加・フック追加
- 「ついでに」行う CLI 改善や UX 変更
