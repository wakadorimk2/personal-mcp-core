# 統合入力UI方針（Issue #187）

> スコープ: ヒートマップ確認 → 候補タグ選択 → 入力/保存 を同一画面で完結させる最小UI実装
> 関連Issue: #187
> 前提文書: [`docs/daily-input-ux-mvp.md`](./daily-input-ux-mvp.md)（レイアウト原則の正本）

---

## 1. 導線統合の方針

### 変更前の状態

| ルート | 内容 |
|---|---|
| `/` | モード切替入力フォーム（ヒートマップなし） |
| `/dashboard` | ヒートマップ＋候補タグ＋入力フォーム（統合済み） |

ユーザーはヒートマップを確認するために `/dashboard` へ、入力のために `/` へと、2画面を行き来する必要があった。

### 変更後の状態

| ルート | 内容 |
|---|---|
| `/`（主導線） | ヒートマップ＋候補タグ＋入力フォーム（統合画面） |
| `/dashboard` | `/` と同一（エイリアス） |
| `/input` | 旧モード切替フォーム（後方互換として保持） |

`/` がデフォルトエントリーポイントとして統合画面を返す。遷移コストを1画面分削減する。

---

## 2. 画面構成

```
[ 草ヒートマップ（直近28日）]   ← 最上部
[ 候補タグ（横並びチップ）   ]   ← ヒートマップ直下
[ テキスト入力欄             ]   ← 候補タグ直下
[ 保存ボタン                 ]   ← 入力欄直下
[ 保存済みサマリー一覧       ]   ← 下部スクロール
```

- ヒートマップ → 候補タグ → 入力 → 保存 の順に上から下へ流れる。
- スクロールなしに候補タグと入力欄の両方が視界に入ることを目標とする（スマホ標準画面高600px前後を想定）。

---

## 3. 候補タグ → 入力欄への連携フロー

1. 候補タグをタップ
2. タグのテキストが入力欄（`#log-text`）に流し込まれる
3. 入力欄がフォーカスされる（スマホではキーボードが開く）
4. 追記・編集してから「保存」をタップ

候補タグ選択と自由記述は排他でない。タップ後に自由記述を追記・編集できる。

候補取得（`/api/candidates`）が失敗した場合は固定候補を表示する。固定候補が空の場合はタグ欄を非表示にする（エラーメッセージは表示しない）。

---

## 4. スマホ操作性の最小要件

| 要素 | 要件 | 実装値 |
|---|---|---|
| 候補タグボタン | タップ可能範囲 ≥ 44px | `min-height: 2.75rem`（≈44px） |
| 保存ボタン | タップ可能範囲 ≥ 44px | `min-height: 2.75rem`、`padding: 0.75rem` |
| 候補タグ文字 | 可読サイズ | `font-size: 1rem`（≈16px） |
| 入力欄 | 最小高さ確保 | `height: 4rem`（≈64px） |
| ページ幅 | スマホ幅対応 | `max-width: 480px; padding: 1rem` |

---

## 5. 実装スコープ外

- `/api/candidates` の詳細実装（候補は現状フォールバック固定値のみ）
- quick モード（モード切替フォーム）の廃止・統合（`/input` に保持）
- アニメーション・レスポンシブ完全最適化などUIポリッシュ
- 候補タップ時の選択済み視覚フィードバック
